** Proyecto guardiaWeb **

[Constitution]

/speckit.constitution Create a lightweight constitution for a simple internal web application that displays an on-call schedule from an Excel spreadsheet.

The constitution should prioritize:
- simplicity over over-engineering
- readability and maintainability
- incremental development
- clean UI and user experience
- minimal dependencies
- fast local development
- pragmatic AI-assisted coding practices

Technical guidelines:
- Use Next.js, React, TypeScript
- Use shadcn/ui for UI components
- Use FullCalendar for calendar views
- Use SheetJS (xlsx) to parse Excel files
- Prefer server-side or local parsing over complex backend services
- Keep architecture simple and modular
- Avoid premature abstraction
- Avoid unnecessary wrappers and utility layers
- Prefer small focused components

Project constraints:
- This is a read-only dashboard application
- No authentication
- No CRUD functionality
- No database unless truly necessary
- No microservices
- No complex infrastructure

Code quality principles:
- Write clear and self-explanatory code
- Prefer simple functions over deep abstractions
- Keep files reasonably small
- Use consistent naming conventions
- Document only non-obvious logic

Testing policy:
- Do NOT generate unit tests
- Focus on manual validation and functional verification during development

AI workflow principles:
- Break implementation into small incremental steps
- Prefer iterative improvements over large one-shot generation
- Always explain architectural decisions before implementing major changes
- Keep prompts and plans concise and pragmatic

Keep the constitution concise and practical, not enterprise-oriented.


[Specify]

/speckit.specify Esta aplicación debe ser una aplicación web interna para visualizar los turnos de guardia de un equipo técnico a partir de una planilla Excel existente.

La aplicación debe permitir consultar de manera visual y sencilla quién está de guardia en cada período del año sin necesidad de abrir o editar la planilla manualmente.

La pantalla principal debe mostrar un calendario o dashboard de guardias con posibilidad de navegar por:
- día
- semana
- mes
- año

Cada período de guardia debe mostrar claramente:
- nombre de la persona asignada
- fecha de inicio
- fecha de fin

La aplicación debe leer automáticamente la información desde un archivo Excel que contiene una fila por cada semana o período de guardia.

El usuario no debe poder modificar datos desde la aplicación. La aplicación es solamente de consulta y visualización.

La experiencia debe ser rápida, simple y clara, pensada para uso interno diario por parte de un equipo técnico.

La interfaz debe incluir:
- navegación temporal simple
- visualización clara de las guardias
- colores o indicadores visuales para diferenciar personas
- diseño moderno y limpio


La aplicación debe permitir:
- cambiar fácilmente entre vistas diaria, semanal, mensual y anual
- buscar o filtrar por persona
- identificar rápidamente quién está de guardia en una fecha determinada

Las guardias activas en la fecha actual deben destacarse visualmente.

El diseño visual debe ser moderno, minimalista y profesional, utilizando una interfaz clara y fácil de leer, evitando sobrecarga visual.

Configuración global (recomendada)
En una terminal (PowerShell o la de Cursor), sustituye el nombre por el que quieras que salga en los commits:

git config --global user.name "Gerardo Fernandez Herrera"
git config --global user.email "lapayota@yahoo.com"
El user.email ya lo tienes; con definir user.name suele bastar:

git config --global user.name "Tu Nombre Apellido"
Comprueba:

git config --global --list | Select-String "user\."
